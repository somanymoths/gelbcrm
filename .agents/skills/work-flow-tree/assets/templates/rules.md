# Правила AI-кодинга

## Общие

* Всегда начинай с короткого плана перед изменениями
* Работай только над текущей секцией из `plan.md`
* Не трогай unrelated code
* Не меняй архитектуру без явной причины
* Предпочитай простые и читаемые решения

## Контроль scope

* Не реализуй весь продукт сразу
* Делай один логический шаг за раз
* Не добавляй "на будущее" то, чего нет в текущем scope
* Если нужно изменить более 5 файлов вне согласованного списка — остановись и пере-согласуй scope

## Тестирование

* Перед реализацией новой фичи определи интеграционные сценарии
* Предпочитай integration / e2e style tests вместо мелких unit tests, если это возможно
* Тестируй пользовательский флоу от начала до конца
* Не удаляй и не ослабляй тесты без причины

## Quality gates (обязательно перед commit)

* Линтер зеленый
* Typecheck зеленый
* Интеграционные тесты зеленые
* Критичный smoke/e2e (если есть) зеленый

## Качество кода

* Держи файлы небольшими
* Предпочитай модульность
* Не создавай огромные универсальные компоненты / сервисы
* Не дублируй код, если можно безопасно вынести общее
* Не вводи абстракции раньше времени

## Отладка

* Если фикс не сработал, не наслаивай костыли
* После неудачной попытки сначала объясни возможные причины
* При сложной ошибке перечисли 3-4 гипотезы до написания нового кода
* Используй логирование, если причина неочевидна

## Безопасное восстановление

* Не применять импульсивный `git reset --hard`
* Сначала делай checkpoint commit в отдельной ветке
* Для отката предпочитай `revert` или возврат к стабильной точке с сохранением истории

## Git-дисциплина

* Один логический шаг = один commit
* После завершения задачи предложи краткий commit message
* Не делай большие несвязанные изменения в одном проходе

## Документация

* Перед использованием новой библиотеки или API сначала прочитай локальную документацию, если она есть
* Если документации нет, не выдумывай API
* При сомнении сначала опиши предположение явно

## Формат ответа

* Сначала: что собираешься сделать
* Потом: какие файлы изменишь
* Потом: сама реализация
* В конце: что проверить вручную и какой commit сделать

## Запрещенное поведение

* Не переписывай рабочий код без необходимости
* Не меняй naming/style сразу во всей кодовой базе
* Не добавляй новые зависимости без явного повода
* Не имитируй уверенность, если не уверен
* Не скрывай побочные изменения

## Протокол завершения

После каждого завершенного шага:

* обнови статус секции в `plan.md`
* перечисли, что было сделано
* перечисли, что проверить руками
* предложи commit message
* остановись и жди следующей задачи
