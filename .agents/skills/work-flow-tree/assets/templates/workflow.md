# Рабочий процесс (work-flow-tree)

Текущий workflow: контекст -> `plan.md` -> секция -> декомпозиция через чат -> реализация по шагам -> gates -> завершение секции -> PR -> пост-фиксация -> деплой.

## 1. Источник правды

* Главный файл управления задачами: `plan.md`.
* Секции поддерживают вложенность: `1`, `1.1`, `1.3.2`.
* Для каждой секции есть отдельный файл: `.workflow/sections/section-<номер>.md`.

## 2. Базовые команды

* `init` — удалить предыдущий workflow проекта и развернуть чистый `.workflow` (или `--keep-existing`, чтобы не удалять).
* `sync-from-source` — обновить шаблоны скилла из `/Users/mxrxzxw/Desktop/moths/workflow`.
* `context-init` — создать/обновить контекст проекта.
* `context-check` — проверить полноту контекста.
* `sync-sections` — синхронизировать секции и ссылки в `plan.md`.
* `sync-sections --dry-run` — показать изменения без записи.
* `start <номер>` — перевести секцию в `in_progress`.
* `finish <номер> done|blocked` — завершить секцию.
* `log "..."` — добавить запись в `progress.md`.
* `pr <url>` — добавить ссылку на Pull Request в `progress.md`.
* `gates` — запустить quality checks.

## 3. Сбор контекста для агента (обязательно после установки skill)

Цель этапа: дать агенту достаточно материала, чтобы он не придумывал продукт на ходу.

Контекст включает:

* `META.md`
* описание пользователей
* JTBD / боли / мотивации
* user flow
* список функций MVP
* ограничения
* референсы
* скриншоты похожих интерфейсов
* бренд-принципы / визуальные правила
* критерии качества

Порядок:

1. Запустить `context-init`.
2. Заполнить файлы в `.workflow/context/`.
3. Запустить `context-check`.
4. Переходить к `start` только после успешного `context-check`.

## 4. Полный цикл по секции

### ✅ Старт

1. Выбрать секцию в `plan.md`.
2. Запустить `start <номер>`.
3. Создать ветку: `section-<номер>-<slug>`.
4. Прогресс-блок этапов выводится только после `start` и только в рамках активной секции.

### 2️⃣ Декомпозиция

Цель этапа: быстро убрать слепые зоны до дизайна и разработки.

Порядок:

1. Агент сначала сам заполняет в `section-<номер>.md`:
   * 3-5 подходов реализации,
   * основные пользовательские сценарии,
   * риски первого релиза,
   * границы MVP и то, что вне MVP,
   * аналоги и UX-паттерны,
   * гипотезы.
2. Затем чат задает вопросы по одному:
   * `Вопрос N`
   * `1. ...`
   * `2. ...`
   * `3. ...`
3. Начиная со второго вопроса чат дает короткое резюме предыдущих ответов.
4. Параллельно агент обновляет в файле секции:
   * цель,
   * задачи,
   * acceptance criteria,
   * техподход,
   * риски,
   * ограничения.

Результат этапа:

* список сценариев,
* список рисков,
* список гипотез,
* понимание рамок MVP,
* техническое задание для реализации.

### 3️⃣ Подтверждение

1. Агент показывает итог декомпозиции.
2. Агент дает ссылку на итоговый `section-<номер>.md`.
3. После подтверждения начинается реализация.

### 4️⃣ Реализация

1. Реализация идет только в scope секции.
2. Изменения делаются шагами (итерациями), без больших невалидированных кусков.
3. Короткие этапы автоматически логируются в `progress.md`.

### 5️⃣ Проверка качества

Минимально обязательно:

* `lint`
* `typecheck`
* `integration tests`

Расширенно (если поддерживается проектом):

* `unit`
* `smoke e2e`
* `build`

Движение дальше только после green по критичным проверкам.

### 6️⃣ Завершение

1. Агент предлагает завершить секцию.
2. После согласия запускается `finish <номер> done|blocked`.
3. Команда обновляет `plan.md`, пишет итог в `progress.md`, генерирует commit message.

### 7️⃣ Пулл-реквест

1. Сделать commit.
2. Открыть PR.
3. Записать ссылку в лог: `pr <url>`.
4. Политика merge: `squash merge`.

### 8️⃣ Пост-фиксация знаний

* Архитектурные решения фиксировать в `decisions.md`.
* Закрытые/повторяемые баги фиксировать в `bugs.md`.
* Не оставлять критичные договоренности только в чате.

### 9️⃣ Деплой

* Деплой выполняется вручную после merge и финальной проверки.
* Перед релизом обязательна ручная продуктовая проверка.

## 5. Главные правила

* Не кодить без контекста.
* Не делать большие куски без промежуточной проверки.
* Не переходить дальше без тестов критичных сценариев.
* Не хранить знания только в голове или чате.
* Не релизить без ручной продуктовой проверки.
* Всегда работать через итерации.
* Относиться к агенту как к усилителю, а не как к автопилоту.
