# План проекта

## 1. Цель

Коротко: что это за продукт и какую проблему он решает.

## 2. Критерии успеха

Что должно быть правдой, если MVP удался.

* Пользователь может зарегистрироваться и войти
* Пользователь может создать основную сущность продукта
* Пользователь может увидеть список своих данных
* Пользователь может изменить или удалить запись
* Основной сценарий работает без ручных обходов
* Ключевые флоу покрыты интеграционными тестами

## 3. Границы MVP

Что входит в первую версию.

* Аутентификация
* CRUD для основной сущности
* Базовый dashboard / список
* Простая валидация форм
* Сохранение данных
* Базовые состояния loading / error / empty

## 4. Вне scope

Что сознательно не делаем сейчас.

* Платежи
* Сложные роли и права
* Уведомления
* Продвинутая аналитика
* Сложная мобильная адаптация
* Микрооптимизации производительности

## 5. Ключевые пользовательские флоу

### Флоу 1. Онбординг

* Пользователь открывает приложение
* Регистрируется / логинится
* Попадает на основной экран

### Флоу 2. Основное действие

* Пользователь создает новую запись
* Видит ее в списке
* Может открыть и изменить

### Флоу 3. Восстановление / edge cases

* Пользователь видит ошибку при невалидных данных
* Пользователь видит пустое состояние, если данных нет
* После перезагрузки данные не исчезают

Ключевые технические решения.

## 8. Ограничения

* Не менять код вне текущей задачи без причины
* Не добавлять новые библиотеки без явной необходимости
* Не рефакторить большие участки без тестов
* Предпочитать простые решения вместо сложных

## 9. Definition of Done (для секции)

Секция считается завершенной только если:

* Acceptance criteria выполнены
* Интеграционные тесты (и обязательные quality gates из `runbook.md`) зеленые
* Ручная проверка пройдена
* Обновлены `plan.md` и `progress.md`
* Сформирован commit message

### Section 1. Project setup

* Статус: todo
* Зависимости: none
* Риски: неверная конфигурация env/test

Цель:
Подготовить проект к разработке.

Задачи:

* Поднять базовую структуру проекта
* Настроить env
* Настроить линтер / форматтер
* Настроить тестовую среду

Acceptance criteria:

* Проект запускается локально
* Тесты запускаются
* Базовая структура понятна

### Section 2. Auth

* Статус: todo
* Зависимости: Section 1
* Риски: ошибки в guard/login flow

Цель:
Дать пользователю вход в систему.

Задачи:

* Login
* Logout
* Guard/private routes
* Error states

Acceptance criteria:

* Пользователь может войти
* Неавторизованный пользователь не видит приватные страницы
* Logout работает корректно

### Section 4. Dashboard / list view

* Статус: todo
* Зависимости: Section 3
* Риски: некорректные empty/error states

Цель:
Показать полезное представление данных.

Задачи:

* Список / карточки

* Loading state
* Error state

Acceptance criteria:

* Список отображается корректно
* Пустое состояние не ломает UI
* Ошибки отображаются понятно
