# GELB CRM: Project Context

Этот файл — единая краткая карта проекта для старта новых задач в отдельных чатах.

## 1) Технологический стек

- Frontend/Backend: Next.js App Router + TypeScript (`src/app/**`, `src/components/**`)
- DB: MySQL (`mysql2`)
- Auth/Session: cookie-сессия (`gelb_session`), role-based доступ
- Основные роли:
  - `admin`: полный доступ
  - `teacher`: ограниченный доступ (в первую очередь журнал/свои сценарии)

## 2) Ключевые домены

- Воронка (`funnel`): карточки, стадии, причины потерь, аудит, отчёты
- Оплаты (`payments`): тарифы, ссылки на оплату, история, webhook-потоки
- Преподаватели (`teachers`): карточки преподавателей, архив, зависимости, привязки
- Студенты (`students`): базовые операции и связь с преподавателями/этапами

## 3) Критичные инварианты (не ломать)

1. RBAC границы:
- `teacher` не должен получать `admin`-доступ к закрытым разделам/эндпоинтам.

2. Payment/Funnel безопасность:
- переходы статусов должны быть явными и идемпотентными;
- не допускать двойной обработки одной и той же бизнес-операции.

3. Изменения БД:
- любые schema-изменения только через новую миграцию в `db/migrations`;
- нельзя менять смысл существующих колонок без одновременного обновления read/write кода.

4. API-контракты:
- без явной задачи не ломать формат ответов, коды ошибок и обязательные поля.

## 4) Где смотреть контекст перед задачей

- API-контракт: `docs/openapi.yaml`
- Funnel-логика: `docs/funnel.md`
- Teachers-правила: `docs/teachers.md`
- Полное ТЗ: `docs/MVP_TECH_SPEC_RU.md`
- БД миграции: `db/migrations/*.mysql.sql`

## 5) Обязательные проверки перед PR

```bash
npm run typecheck
npm run lint
npm run build
```

Если задача затрагивает БД:
- проверить, что есть новая миграция;
- проверить применение миграции на целевом окружении.

## 6) Деплой (операционный контур)

- Команда: `npm run deploy`
- Скрипт: `scripts/deploy.sh`
- В процессе выполняются:
  - sync кода на сервер;
  - `npm ci`;
  - `npm run build`;
  - `npm run db:migrate`;
  - restart сервиса (`systemd`/`pm2`);
  - healthcheck `/api/health`.

## 7) Шаблон старта нового чата (одна задача)

Используйте этот блок в первом сообщении нового чата:

```md
Задача: <краткая цель>
Ветка: codex/<feat|fix|chore>/<issue>-<slug>
Scope: <что меняем>
Не трогать: <что нельзя менять>
Definition of Done:
- [ ] typecheck/lint/build зелёные
- [ ] RBAC не нарушен
- [ ] (если БД) миграция добавлена и проверена
Релевантные источники:
- docs/openapi.yaml
- docs/funnel.md
- docs/teachers.md
```

## 8) Правило контекста

Контекст проекта хранится в:
- документах `docs/*`;
- PR-описаниях;
- миграциях и коде.

Чат — только рабочая сессия под одну задачу.
