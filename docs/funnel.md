# Раздел CRM: Воронка

## 1. Описание функционала

### 1.1 Цель
Раздел `Воронка` предназначен для:
1. Управления продажами (лиды/ученики по этапам).
2. Учёта фактической выручки на основе реальных оплат (без вероятностного прогноза).

### 1.2 Роли и доступ
На первом этапе поддерживается одна роль:
1. `Администратор`:
- Полный доступ к разделу.
- Настройка этапов (в рамках доступного справочника/конфига).
- Создание, редактирование, архивирование, восстановление карточек.
- Просмотр логов и отчётов.
- Экспорт отчётов.

### 1.3 Основная сущность
1. На ранних этапах сущность имеет тип `Лид`.
2. При переводе в этап `На занятиях` или `Последнее занятие` сущность становится `Ученик`.
3. Обратная конверсия `Ученик -> Лид` запрещена.
4. Карточка остаётся одной и той же (без создания второй карточки).

### 1.4 Этапы воронки
Порядок этапов:
1. `Заинтересовался`
2. `Квалификация`
3. `Знакомство`
4. `Оплата`
5. `На занятиях`
6. `Последнее занятие`
7. `Отказался`
8. `Перестал заниматься`

Категории этапов:
1. Успешная конверсия: `На занятиях`, `Последнее занятие`.
2. Потеря: `Отказался`, `Перестал заниматься`.

### 1.5 Формат доски
1. Одна горизонтальная Kanban-доска.
2. Колонки соответствуют этапам воронки.
3. Смена этапа доступна:
- Через drag-and-drop карточки.
- Через изменение этапа в карточке.

### 1.6 Сортировка карточек в колонке
Правило сортировки по умолчанию:
1. Если у карточки есть `дата следующего занятия` -> сортировать по возрастанию (ближайшие сверху).
2. Иначе -> сортировать по `дате создания` по убыванию (новые сверху).

### 1.7 Поля карточки
Обязательные поля карточки:
1. `Фамилия и имя ученика`
2. `Телефон`
3. `Контакт`
4. `Email`
5. `Источник лида`
6. `Комментарий`
7. `Дата начала занятий`
8. `Дата последнего занятия`
9. `Осталось оплаченных занятий`

Дополнительные рабочие поля/состояния:
1. `Преподаватель`
2. `Дата следующего занятия`
3. `Статус оплаты`
4. `Причина потери` (обязательно для этапов потери)

### 1.8 Что видно на карточке на доске
1. `Фамилия и имя ученика`
2. `Преподаватель (Фамилия и имя)`
3. `Дата следующего занятия`
4. `Осталось оплаченных занятий`

### 1.9 Действия в карточке
1. Назначить преподавателя.
2. Добавить комментарий.
3. Просмотреть лог (история изменений).
4. Создать ссылку на оплату.

### 1.10 Назначение преподавателя
1. Только вручную администратором.
2. Из списка существующих преподавателей.

### 1.11 Комментарии
1. Поддерживается общая лента комментариев в карточке.
2. Поддерживаются комментарии, привязанные к конкретному этапу.

### 1.12 Логирование и аудит
1. Обязателен полный аудит изменений:
- Кто изменил.
- Когда изменил.
- Что было изменено.
2. Логируются все изменения полей и этапов.

### 1.13 Оплаты и ссылка YooKassa
1. Ссылка на оплату генерируется автоматически через интеграцию с `YooKassa`.
2. При генерации выбирается тариф из существующего раздела `Оплаты`.
3. Статусы оплаты в карточке:
- `Ожидает оплату`
- `Оплачено`
- `Ошибка/отклонено`
- `Истекла`

### 1.14 Архивирование
1. Физическое удаление запрещено.
2. Доступно только архивирование карточки.
3. Из архива можно восстановить карточку с выбором этапа.

### 1.15 Потери
1. Для перехода в этапы `Отказался` и `Перестал заниматься` обязательна причина из справочника.
2. Показатель потерь в отчётах считается как текущее количество карточек в этих этапах.

### 1.16 Создание карточек и дубли
1. В первой версии карточки создаются только вручную администратором.
2. Проверка дублей обязательна при создании:
- По `телефону`.
- По `email`.

### 1.17 Фильтры, поиск, уведомления
1. Поиск и фильтры в первой версии не реализуются.
2. Уведомления и напоминания в первой версии не реализуются.

### 1.18 Отчётность (MVP)
Обязательные отчёты:
1. Количество лидов/учеников по этапам.
2. Потери: текущее количество в этапах `Отказался` и `Перестал заниматься`.

## 2. Техническое задание

### 2.1 Границы реализации
В рамках задачи реализуются:
1. UI доски и карточки.
2. Backend API для операций воронки.
3. Интеграция генерации платёжной ссылки с YooKassa.
4. Получение тарифов из раздела `Оплаты`.
5. Миграции БД.
6. Аудит действий.
7. Архив/восстановление.
8. Базовые отчёты и экспорт.

### 2.2 Бизнес-правила
1. Этапы строго соответствуют списку из раздела 1.4.
2. Перевод карточек между этапами возможен вручную двумя способами (drag-and-drop и в карточке).
3. Stage gating (жёсткая валидация перед переходом) не применяется.
4. Сущность становится `Ученик` при первом попадании в этапы `На занятиях` или `Последнее занятие`.
5. После смены типа на `Ученик` возврат к типу `Лид` запрещён.
6. При переводе в `Отказался` или `Перестал заниматься` причина потери обязательна.
7. Карточка не удаляется физически, только архивируется.
8. При восстановлении из архива этап выбирается администратором вручную.
9. Проверка дублей по телефону/email выполняется при создании карточки.
10. Прогноз/выручка в разделе учитывает только фактически подтверждённые оплаты.

### 2.3 User Stories
1. Как администратор, я хочу создавать карточку лида вручную, чтобы вести нового клиента по этапам.
2. Как администратор, я хочу перемещать карточку между этапами drag-and-drop, чтобы быстро обновлять состояние.
3. Как администратор, я хочу менять этап из карточки, чтобы управлять воронкой без перетаскивания.
4. Как администратор, я хочу назначать преподавателя из списка, чтобы закреплять ученика.
5. Как администратор, я хочу генерировать ссылку YooKassa по выбранному тарифу, чтобы отправлять клиенту оплату.
6. Как администратор, я хочу видеть текущий статус оплаты, чтобы понимать состояние платежа.
7. Как администратор, я хочу вести общие и этапные комментарии, чтобы фиксировать контекст работы.
8. Как администратор, я хочу видеть полный аудит изменений, чтобы понимать историю действий.
9. Как администратор, я хочу архивировать карточку и восстанавливать её с выбором этапа, чтобы контролировать жизненный цикл без удаления данных.
10. Как администратор, я хочу видеть отчёт по количеству карточек на этапах и потерям, чтобы оценивать состояние воронки.

### 2.5 API (REST)

Базовый префикс: `/api/funnel`

#### 2.5.1 Справочники
1. `GET /stages` -> список этапов.
2. `GET /loss-reasons` -> список причин потери.
3. `GET /teachers` -> список преподавателей для назначения.
4. `GET /payment-tariffs` -> тарифы из раздела `Оплаты`.

#### 2.5.2 Карточки
1. `GET /board` -> данные доски: этапы + карточки по этапам.
2. `POST /cards` -> создание карточки вручную.
3. `GET /cards/{id}` -> детальная карточка.
4. `PATCH /cards/{id}` -> редактирование полей карточки.
5. `PATCH /cards/{id}/stage` -> смена этапа.
6. `PATCH /cards/{id}/teacher` -> назначение/смена преподавателя.
7. `POST /cards/{id}/archive` -> архивировать.
8. `POST /cards/{id}/restore` -> восстановить с указанием этапа.

#### 2.5.3 Комментарии
1. `GET /cards/{id}/comments` -> общие + этапные комментарии.
2. `POST /cards/{id}/comments` -> добавить комментарий (`stage_id` nullable).

#### 2.5.4 Оплаты
1. `POST /cards/{id}/payment-links` -> создать ссылку YooKassa по выбранному тарифу.
2. `GET /cards/{id}/payment-links` -> список ссылок и статусов.
3. `POST /integrations/yookassa/webhook` -> входящий вебхук обновления статуса.

Маппинг статусов YooKassa -> статусы карточки:
1. `pending` -> `Ожидает оплату`
2. `succeeded` -> `Оплачено`
3. `canceled`/`failed` -> `Ошибка/отклонено`
4. истечение TTL ссылки -> `Истекла`

#### 2.5.5 Аудит и отчёты
1. `GET /cards/{id}/audit` -> полный аудит карточки.
2. `GET /reports/stage-counts` -> количество карточек по этапам.
3. `GET /reports/losses-current` -> текущее количество карточек в loss-этапах.
4. `GET /reports/export?type=stage-counts|losses-current&format=csv|xlsx` -> экспорт.

### 2.6 UI/UX требования

#### 2.6.1 Экран `Воронка`
1. Горизонтальная Kanban-доска по этапам.
2. В каждой колонке:
- Заголовок этапа.
- Счётчик карточек.
- Список карточек с полями из п.1.8.
3. Поддержка drag-and-drop между колонками.
4. По клику карточка открывается в детальном окне/панели.

#### 2.6.2 Детальная карточка
Разделы:
1. Основные данные (редактируемые поля).
2. Назначение преподавателя.
3. Оплаты (создание ссылки YooKassa, текущие статусы).
4. Комментарии:
- Общая лента.
- Комментарии по этапам.
5. Аудит (лог изменений).

#### 2.6.3 Экран `Архив`
1. Список архивных карточек.
2. Действие `Восстановить`.
3. При восстановлении обязательный выбор этапа.

#### 2.6.4 Экран `Отчёты`
1. Виджет/таблица количества карточек по этапам.
2. Виджет/таблица потерь (текущие значения для `Отказался` и `Перестал заниматься`).
3. Кнопка экспорта CSV/XLSX.

### 2.7 Сценарии (E2E)

1. Создание карточки:
- Админ вводит обязательные поля.
- Система проверяет дубли по телефону/email.
- При отсутствии дублей карточка создаётся в этапе `Заинтересовался`.

2. Перевод по этапам:
- Админ перетаскивает карточку в новый этап или меняет этап в карточке.
- Система сохраняет изменение и пишет запись в аудит.

3. Потеря лида/ученика:
- При переводе в `Отказался`/`Перестал заниматься` система требует причину из справочника.
- После выбора причины переход сохраняется, событие фиксируется.

4. Конверсия в ученика:
- При первом переходе в `На занятиях`/`Последнее занятие` `entity_type` переключается на `student`.
- Возврат к `lead` запрещён.

5. Назначение преподавателя:
- Админ выбирает преподавателя из списка.
- Система сохраняет связь и пишет аудит.

6. Генерация оплаты:
- Админ выбирает тариф из раздела `Оплаты`.
- Система создаёт платёж в YooKassa, сохраняет ссылку, выставляет статус `Ожидает оплату`.
- После webhook статус обновляется (`Оплачено`/`Ошибка/отклонено`/`Истекла`).

7. Архив/восстановление:
- Админ архивирует карточку, она исчезает с доски.
- В архиве админ запускает восстановление и выбирает этап.
- Карточка возвращается в соответствующую колонку.

### 2.8 Критерии приёмки
1. Воронка отображается как горизонтальная доска с 8 этапами в заданном порядке.
2. Карточка перемещается между этапами drag-and-drop и через редактирование в карточке.
3. Сущность необратимо меняется с `Лид` на `Ученик` при достижении этапов успеха.
4. Для этапов потери нельзя сохранить переход без выбора причины из справочника.
5. Создание карточки блокируется при дубле телефона или email.
6. Назначение преподавателя доступно только вручную администратором.
7. Ссылка оплаты создаётся через YooKassa по выбранному тарифу из раздела `Оплаты`.
8. В карточке корректно отображаются статусы: `Ожидает оплату`, `Оплачено`, `Ошибка/отклонено`, `Истекла`.
9. В карточке доступны общие комментарии и комментарии по этапам.
10. Полный аудит изменений доступен в карточке.
11. Физическое удаление карточек отсутствует; работает архивирование и восстановление с выбором этапа.
12. В отчётах доступны:
- Количество карточек по этапам.
- Текущие потери по двум этапам.
13. Поиск, фильтры и уведомления отсутствуют в первой версии.

### 2.9 Нефункциональные требования
1. Все операции изменения данных должны быть атомарными и журналироваться в `funnel_audit_log`.
2. API должно возвращать понятные коды ошибок валидации (`409` для дублей, `422` для бизнес-ограничений).
3. Все даты/время хранить в UTC (`timestamptz`), отображать в таймзоне школы/пользователя.
4. Интеграция с YooKassa должна использовать подпись/проверку webhook и идемпотентную обработку событий.

### 2.10 Ограничения MVP (явно не входит)
1. Поиск и фильтры по воронке.
2. Автоматические уведомления и напоминания.
3. Импорт из CSV/XLSX.
4. Автозагрузка лидов из сайта/API/вебхуков.
5. Вероятностный прогноз (взвешенный pipeline forecast).
